## Project Claude Rules

Two rule files exist and are **byte-identical**: `CLAUDE.md` and `claude.md`. (`task.md` and
`bet.md` are *aspirational feature specs* the operator has fed in, not the standing rules ŌĆö see
Gotchas.) The must-follow constraints:

- [ ] **Build as a sequence of working milestones** ŌĆö repo architecture ŌåÆ config ŌåÆ data schema ŌåÆ
  leakage-safe feature pipeline ŌåÆ Elo baseline ŌåÆ Poisson/DixonŌĆōColes ŌåÆ chronological backtesting ŌåÆ
  calibration ŌåÆ correct-score probabilities.
- [ ] **Ship real code, tests, and docs** ŌĆö never just design/pseudocode.
- [ ] **Synthetic data must be isolated and clearly labelled** ŌĆö never present synthetic results as
  real model performance. (In practice all data is real Football-Data.co.uk.)
- [ ] **Per-milestone workflow, every time:** 1) implement ŌåÆ 2) run its tests ŌåÆ 3) fix errors ŌåÆ
  4) **record commands and results** ŌåÆ 5) continue to the next milestone.
- [ ] **Stop conditions** ŌĆö pause and defer to the user only when an action requires **credentials,
  payment, external deployment, or irreversible access**. Otherwise keep going.
- [ ] **Never claim a scoreline (or outcome) is certain.** Always return full probability
  distributions and *calibrated* probabilities.

Additional discipline the codebase enforces (binding conventions): calibration over raw accuracy;
**strictly pre-kickoff features (no leakage)**; **chronological/walk-forward evaluation only, never
random splits**; the de-vigged bookmaker line is the strong baseline to be honest about (the ensemble
does *not* beat the market on log-loss and lacks significance tests).

**Surfaced in the app (this session):** `GET /api/decisions?league=...` in `app.py` scores upcoming
fixtures via the existing `Predictor`/cache path (no retrain, no duplicated feature logic), attaches
the **real pre-kickoff feed odds** (1X2 always; O/U 2.5 via `data.parse_fixture_totals_odds` when the
feed carries it), and returns `{"decisions": [...], "summary": {...}}`. The **Value Scan** tab in
`templates/index.html` renders it: three side-by-side views (pure / market / hybrid), per-selection
price comparisons (offered odds, break-even, fair/model/conservative probability, edge in points, raw
and conservative EV), a No-bet/Watchlist/Qualified/Strong-evidence badge, rejection-reason codes, top-5
scorelines, and a research summary ŌĆö behind a visible advisory banner, no certainty language, no
staking UI. `python -m scripts.research_decisions --days N` runs the same path from the CLI and writes
`outputs/decisions/research.json`.

**Honest odds timestamp (this session):** the fixtures feed carries no per-row odds timestamp, so
`/api/decisions` and `research_decisions.py` pass the feed's **verifiable observed-at time** (the last
fixtures-sync time, else the `fixtures.csv` mtime) as `odds_timestamp`, with `decision_time = now`.
This is non-leaking (`odds_timestamp Ōēż now < kickoff`) and lets `check_odds_timestamp` judge staleness
honestly instead of rejecting every price with a blanket `ODDS_STALE`. On a freshly-synced feed the
remaining rejections are real signal (`INSUFFICIENT_HISTORICAL_SAMPLE` ŌĆö no per-market validation
profiles loaded yet ŌĆö plus `NO_POSITIVE_CONSERVATIVE_EV` / `EDGE_BELOW_THRESHOLD`). "No bet" stays the
honest default; nothing here demonstrates a positive edge.

**Market-validation profiles wired into serving (this session):** a market is only evaluated
once it carries a passed-quality out-of-time profile ŌĆö otherwise every selection is rejected
`INSUFFICIENT_HISTORICAL_SAMPLE` (the honest default). `scripts/build_market_profiles.py` runs the
leakage-safe walk-forward decision backtest (profile fitted on the **validation** window only, never
the test set) and writes `artifacts/market_profiles.json` via `decisions.save_profiles`. Serving
(`/api/decisions`, `scripts/research_decisions.py`) loads it with `decisions.load_profiles` and threads
it through `decide_for_fixtures(..., market_profiles=...)`; a missing file simply keeps every market
unvalidated (no crash). `/api/decisions` also returns `markets_validated`. Whether a market *passes*
depends on its real out-of-time log-loss/ECE ŌĆö the ensemble still does not beat the market baseline, so
qualifying selections remain rare and "No bet" stays the default.

**Bet modes surfaced (this session):** `GET /api/bet-modes?mode=smart|high-return|compare[&league=ŌĆ”]`
in `app.py` reuses the live `Predictor` + feed odds via `decisions.modes_for_fixtures` (which builds
each fixture's `MatchDecision` once, captures the pure-model probability per selection, then
re-thresholds under each mode) ŌĆö no retrain, no client odds, no bet placed. The **Value Scan** tab in
`templates/index.html` has an "All markets / Smart Bet / High Return / Compare" selector: each mode
renders per-fixture cards with the 0-100 score, its component bars, the price/probability grid, the
stress-test EVs, a status badge, risk warnings and rejection codes; Compare shows both modes side by
side. `python -m scripts.research_decisions --bet-mode smart|high-return|compare --days N` runs the same
path from the CLI (writes `outputs/decisions/research.<mode>.json`). `python -m scripts.backtest_bet_modes
--mode smart|high-return|both` runs the chronological per-mode backtest. Because the ensemble does not
beat the de-vigged market, both modes still emit "No bet" for essentially every current fixture ŌĆö the
honest result; the value is in showing *why* each candidate is rejected, not in manufacturing picks.
High-return selections are labelled as usually LOSING more often than they win (value is in the price,
not the hit rate).

**Tools console (this session):** a **Tools** tab + `GET /api/tools` / `POST /api/tools/run` let the
operator run project scripts from the UI. `TOOLS` in `app.py` is a **whitelist** of scripts (research
decisions, backtest bet modes, build market profiles, backtest decisions, backtest models, predict
upcoming, make report, paper-bet report) each with **typed params** (int/date/text/choice) rendered as a
form, plus an optional advanced-args field. `_build_argv` validates params and `_script_job` runs
`python -m <module> ŌĆ”` as a **subprocess with NO shell** (argv only; advanced args are `shlex`-tokenised,
never a shell string), streaming stdout line-by-line into the existing single-job `JobManager` and the
shared progress panel. Only one job runs at a time. Train and data-sync are deliberately excluded (their
dedicated endpoints hot-reload the served model / record sync metadata); everything advisory stays
research-only and places no bets.

**Paper-bet ledger analytics (this session):** `match_predict/bet_analytics.py`
(`performance_report()` / `report_from_store()`) reports **realized return** (P&L, ROI, win rate, max
drawdown, longest losing run, return by odds band) kept strictly separate from **forecast quality**
(Brier + log-loss + a reliability table over the `model_prob` actually staked vs. realized wins),
reusing `evaluation.metrics.log_loss`. Every figure carries its sample size; `<30` settled bets are
flagged as not meaningful. Surfaced at `GET /api/bet_report` and in a **Performance & calibration**
panel under the My Bets tab; CLI `python -m scripts.bet_report`. Voids excluded from returns; advisory
paper-money only ŌĆö a positive ROI is never presented as a proven edge.

**Honest caveat, unchanged by this surfacing:** the ensemble does not beat the de-vigged market
baseline on log-loss (no significance tests exist), and on the current live fixtures feed almost every
selection resolves to **NO BET** because the feed carries no captured odds timestamp ŌĆö `ODDS_STALE`
is the expected, conservative default, not a bug.

## Data flow

**Training / batch prediction (offline):**
```
sync (data/sync.py: all|latest|fixtures)  ŌåÆ  CSVs on disk (.data-cache zips cached)
  ŌåÆ ingest (load_all)  ŌåÆ  canonical frame
  ŌåÆ validate           ŌåÆ  ValidationReport
  ŌåÆ features (EloŌåÆformŌåÆcontextŌåÆmarket, all shift(1)/pre-match)
  ŌåÆ models: DixonColes (score matrix) ┬Ę GBM (P H/D/A) ┬Ę MarketBaseline (de-vig P)
  ŌåÆ ensemble (multinomial logit on log-probs)  ŌåÆ  P_ens
  ŌåÆ calibration (temperature/isotonic)         ŌåÆ  P_final
  ŌåÆ reconcile: rescale DC matrix to P_final marginals
  ŌåÆ MatchPrediction: 1X2, matrix xG, top-5 scores, BTTS, O/U, AH, team totals,
    confidence, uncertainty, SHAP, reasons
```
Two ensembles are trained: full (market+GBM+DC) and a **market-free fallback** (GBM+DC) used when a
fixture has no odds. Evaluation (`WalkForwardBacktest`) does the same strictly chronologically.

**Web fixtures ŌåÆ predict ŌåÆ bet ŌåÆ settle (`app.py` + `store.py` + `betting.py`):**
```
sync fixtures  ŌåÆ  football-data/fixtures.csv  ŌåÆ  parse_fixtures()
  ŌåÆ predict_fixtures_cached(): covered leagues scored via Predictor; cached in fixture_cache
    keyed by (trained_through, match_id); live feed odds re-attached fresh each call
  ŌåÆ UI 1/X/2 odds buttons  ŌåÆ  POST /api/bet
  ŌåÆ betting.bet_from_fixture() (odds from feed only)  ŌåÆ  Store.place_bet() (debits wallet)
  ŌåÆ next sync lands fresh results  ŌåÆ  _settle_now() ŌåÆ betting.settle_open_bets()
    ŌåÆ Store.settle_bet(): pays stake├Śodds on win, returns stake on void; updates balance/P&L/ROI
```
Retrain (`/api/train`) ŌåÆ `train_and_save` ŌåÆ `load_predictor()` hot-reload ŌåÆ `STORE.clear_cache()`.

## Conventions

- **League labels:** stable codes from `data/schema.py:LEAGUE_BY_DIV` (`England-PL`, `France-L1`, ŌĆ”);
  pretty names in `app.py:LEAGUE_LABELS`, graceful fallback to the raw code. `DIV_TO_FOLDER` keeps
  on-disk paths in sync with labels.
- **Team names:** Football-Data spelling exactly (`Man City`, `Nott'm Forest`, `Paris SG`); fixtures
  reuse the same spelling so names line up with history. No alias resolver implemented.
- **File layout:** history `football-data/<country>/<League>_YYYY-YYYY.csv`; hold-out flat in `testing/`;
  fixtures `football-data/fixtures.csv`.
- **match_id:** `league|season|YYYYMMDD|home|away` (upcoming fixtures append `|UPCOMING`). July-rollover seasons.
- **Odds preference:** B365 ŌåÆ Avg ŌåÆ BW ŌåÆ WH ŌåÆ PS. Closing odds (PSC*) ingested but leakage-sensitive.
- **Testing:** pytest, property/invariant-driven, no network (sync tests stub HTTP). `pip install -e ".[dev]"`
  then `pytest -q`. Optional `FEAT_CACHE`/`TRAIN_CACHE`/`--cache` pickles the feature frame.
- **Progress / background jobs:** `JobManager` runs one job at a time (sync/train/evaluate/settle), a
  `progress(msg, pct)` callback threads through; UI polls `/api/job/status`; log is a bounded 400-line deque.
- **Persistence:** models via `joblib`; history via parquet if `pyarrow` else `.pkl`. SQLite DB is fully
  derivable from CSVs + bundle.

## Gotchas / stop-conditions

**Require the user (per CLAUDE.md):** credentials, payment, external deployment, or irreversible access.
**Real bookmaker integration / real-money betting is out of scope** ŌĆö the betting layer is paper-money
only (Ōé¼1000 play balance); no auto placement, no staking systems, no martingale, no loss-chasing.

**Spec-vs-reality:** `task.md` and `bet.md` are large **aspirational specs** ŌĆö treat as backlog, not
current state. `ARCHITECTURE.md` and `README.md` accurately describe what exists. `features.md` documents
design intent (some features marked `(design)` are not built).

**Known fragilities:**
- **SSL trust:** `data/sync.py:_fetch` tries certifi and, on a broken local CA store, falls back to an
  unverified context for the football-data.co.uk host only. Expected on python.org macOS builds ŌĆö don't
  turn it into a hard failure.
- **Off-season / stale fixture feed:** `sync_fixtures` tolerates the current season not yet published and
  an empty/absent fixtures file. Around the July rollover it refreshes both the calendar season and the
  season(s) the fixtures' dates imply.
- **Buffered logs / single-job manager:** only one background job at a time (a second returns HTTP 409);
  log is a bounded 400-line deque; progress `pct` is best-effort. Foreground `nohup` logs are block-buffered.
- **Honest-metrics discipline:** the ensemble does **not** beat the market baseline on log-loss and there
  are **no paired bootstrap CIs / significance tests yet**. Never upgrade numerical orderings into "beats
  the market" claims. Ensemble "influence" values are normalized coefficient magnitudes, **not** mixture weights.
- **Cache invalidation:** fixture predictions are cached by `trained_through`; always `STORE.clear_cache()`
  after retraining (wired in `/api/train`).
- **Port 5000** is taken by macOS AirPlay; the app defaults to **5001** (`PORT=` overrides).
